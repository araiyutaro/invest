## Spacing Scale

This is a legacy inline-style HTML generator using `rem` units directly on individual elements (no design-token file, no 4px/8px grid system exists in the codebase). This phase does **not** introduce a new spacing system — it reuses the exact `rem` values already established by the "社名一致" badge precedent, scaled to `font-size` per Code Examples in `22-RESEARCH.md`.

| Token | Value | Usage |
|-------|-------|-------|
| badge-padding-v | 0.15rem (~2.4px) | Vertical padding inside both new badges (matches "社名一致" precedent exactly) |
| badge-padding-h | 0.5rem (~8px) | Horizontal padding inside both new badges (existing precedent uses 0.4rem; new badges use 0.5rem per Research Code Examples — both are established, no invention) |
| badge-gap | 0.5rem (~8px) | `margin-left` separating each badge from the symbol text / preceding badge in the `<h4>` |
| card-padding | 1.2rem (~19.2px) | `.agent-card` padding — **unchanged**, not touched by this phase |

Exceptions: This report generator does not follow a strict 4px-multiple grid (it predates any token system and uses `rem`-based inline styles throughout, e.g. `0.15rem`, `0.4rem`, `0.85rem`). This phase intentionally continues that convention rather than introducing a parallel px-grid system, to keep the two new badges visually and structurally consistent with the "社名一致" badge already shipping in the same card. Do not retrofit the rest of the report to a 4px grid as part of this phase — out of scope.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f0f1a` | Page background (`body`, unchanged) |
| Secondary (30%) | `#1e1e2e` | Card surfaces (`.agent-card`, `.news-card`, unchanged) |
| Accent (10%) | `#10b981` | Portfolio report's semantic accent (green) — `generateBaseStyles("#10b981")` call site in `generatePortfolioReportHtml`. Reserved for: `<h1>` bottom border, `<h2>` left border + text color, and the `保持` (hold) decision's `border-left`/text color. **Unchanged by this phase** — badges must never override this. |
| Destructive (urgent badge) | `#ef4444` | Reserved for: (1) urgent badge background (`⚠ 緊急`, D-16), (2) `全売却` decision `border-left`/text color (existing, unchanged). White text (`#fff`) on this background for the badge. |
| Warning (decision-changed badge) | `#f59e0b` | Reserved for: (1) decision-changed badge background (`判断変更: ... → ...`, D-17), (2) `一部売却` decision `border-left`/text color (existing), (3) `riskNote` text color (existing, unchanged). Dark text (`#1a1a28`) on this background for the badge — **not** white, since amber has poor contrast with white (per Research Code Examples). |

Accent reserved for: `<h1>` border-bottom, `<h2>` border-left + text color, `保持` decision indicator only. **Never** used for the new urgent/changed badges — those use the two semantic colors above, per D-18 (`border-left` decision color is never overridden by badge presence).

**D-18 compliance (critical, non-negotiable):** `border-left-color` on `.agent-card.news-card` is computed **exclusively** by `decisionColor(h.decision)` (保持=`#10b981`/買増=`#3b82f6`/一部売却=`#f59e0b`/全売却=`#ef4444`) and must remain unaffected by `urgent` or `decisionChanged` state. Badges are additive, rendered inside `<h4>`, never touching `border-left`.

**Claude's Discretion resolved — urgent card tint (D-18):** No additional card-level background tint or box-shadow is added for `urgent === true` cards beyond the red badge itself. Rationale: the `⚠ 緊急` badge (bold white-on-red pill, prefixed with a warning glyph) already provides strong visual salience inside the `<h4>`; adding a second visual layer (e.g. a red background tint across the whole card) risks visually competing with — and potentially being misread as overriding — the `border-left` decision-color semantics that D-18 explicitly protects. Minimal-diff, single-signal-per-concern is preferred and matches this codebase's existing minimalism (the "社名一致" badge also has no accompanying card-level treatment).

## Component Specifications (phase-specific addendum)

Exact, load-bearing implementation contract for the two new badge helpers extending `formatHoldingEvaluationsHtml` in `src/scripts/generate-portfolio-report.ts` (lines 47-69). Sourced verbatim from `22-RESEARCH.md` Code Examples, cross-checked against D-16/D-17/D-18.

```typescript
function formatUrgentBadgeHtml(urgent: boolean): string {
  if (!urgent) return "";
  return ` <span style="display:inline-block;background:#ef4444;color:#fff;font-size:0.75rem;font-weight:bold;padding:0.15rem 0.5rem;margin-left:0.5rem;border-radius:999px;">⚠ 緊急</span>`;
}

function formatDecisionChangedBadgeHtml(
  decisionChanged: boolean | undefined,
  previousDecision: string | undefined,
  decision: string,
): string {
  if (decisionChanged !== true) return ""; // undefined AND false both render nothing (D-14)
  return ` <span style="display:inline-block;background:#f59e0b;color:#1a1a28;font-size:0.75rem;font-weight:bold;padding:0.15rem 0.5rem;margin-left:0.5rem;border-radius:999px;">判断変更: ${escapeHtml(previousDecision ?? "?")} → ${escapeHtml(decision)}</span>`;
}
```

Insertion point inside the existing `<h4>` (card header), badge order is urgent-first then decision-changed, both before the existing `float:right` decision-color span:

```html
<h4>${escapeHtml(h.symbol)}${nameJa}${formatUrgentBadgeHtml(h.urgent)}${formatDecisionChangedBadgeHtml(h.decisionChanged, h.previousDecision, h.decision)}
  <span style="float:right;color:${color};font-weight:bold;">${escapeHtml(h.decision)}</span></h4>
```

**Visual states to verify (executor + checker):**
1. Neither urgent nor changed — card renders exactly as today (regression baseline).
2. `urgent: true` only — red `⚠ 緊急` pill appears, `border-left` color still driven by `decision` (not overridden to red).
3. `decisionChanged: true` only — amber `判断変更: X → Y` pill appears, `border-left` color still driven by current `decision`.
4. Both `urgent: true` and `decisionChanged: true` — both pills render side by side (urgent first), no layout collision, `border-left` still driven by `decision` alone.
5. `decisionChanged: undefined` (no prior-day snapshot, or symbol not in prior snapshot) — no badge, no error, visually identical to state 1 (D-14: undefined ≠ false, but both render nothing).
6. Fallback path (`portfolioAnalysis === null`) — no cards render at all; badges are structurally unreachable here (D-21), confirm no regression to the existing fallback message.
