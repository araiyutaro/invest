## Fixed Issues

### CR-01: Checksum-lookup `grep -F | awk` pipeline can silently crash `run.sh` before it reports the real exit code

**Files modified:** `scripts/run.sh`
**Commit:** ddeb531
**Applied fix:** Guarded the `EXPECTED=$(grep -F ... | awk ...)` command substitution with `2>/dev/null` and `|| true`, matching the same pattern already used for the sibling `FAILED_STEP=` extraction later in the file. This prevents `pipefail`/`errexit` from aborting the entire script when `grep -F` finds no matching checksum line (exit status 1) for a protected file that appeared after the checksum snapshot was recorded. Verified with an isolated reproduction of the exact scenario described in the review (file created mid-run after checksum recording): before the fix the script died silently at that line with exit 1 and never reached `=== reached end of script ===`; after the fix the script completes normally with `EXPECTED` empty and exit 0.

### WR-01: Unguarded `git checkout --` in the same HTML-restore block can also abort the script under `set -e`

**Files modified:** `scripts/run.sh`
**Commit:** 17f283c
**Applied fix:** Wrapped `git -C "$PROJECT_DIR" checkout -- "$f"` in an `if`/`else` so a restore failure no longer triggers `set -e` termination. On success, behavior is unchanged (file added to `$RESTORED`, success message logged). On failure, a new `HTML保護: $f の復元に失敗しました` message is logged via `tee -a "$LOG_FILE"` and the script continues to the finish banner, exit-code notification, and log rotation — matching the fix suggested in the review.

### WR-02: Deploy failure notification message is now inaccurate for 2 of its 3 possible causes

**Files modified:** `.claude/commands/invest.md`
**Commit:** 4b85e62
**Applied fix:** Replaced the hardcoded `git push に失敗` failure message in the deploy step's failure-branch instructions with a generalized message covering all three actual failure causes introduced by this phase (date-validation, `git commit`, `git push`): `デプロイ処理が失敗（date検証/commit/pushのいずれか、詳細はログのconsole.error出力を参照）`, and the corresponding `[PIPELINE:FAIL]` line was updated to match. This restores diagnostic accuracy for the `FAILED_STEP` notification feature this same phase built, directing the user to the specific `console.error` output already produced by the `node -e` deploy script for root-causing.
