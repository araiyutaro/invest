## Summary

This phase's stated goal was to make `scripts/run.sh`'s failure/exit-code reporting actually reliable (OPS-01/OPS-03) and to close a shell-injection gap plus an observability gap in `.claude/commands/invest.md`'s deploy step (WR-04/D-08). The targeted fixes themselves are implemented correctly:

- The `EXIT_CODE=0` + `... || EXIT_CODE=$?` idiom correctly replaces the old `|| true` + stale `PIPESTATUS[0]` masking bug (verified via isolated reproduction).
- `PROTECT_FILES` is now a proper bash array iterated with quoted expansion, and the checksum lookup uses `grep -F` for fixed-string matching (WR-01/WR-03 resolved as specified).
- The deploy step's `date` value is validated against `/^\d{4}-\d{2}-\d{2}$/` before touching git, and `git commit`/`git push` were correctly migrated to `spawnSync` with argument arrays (no shell interpolation), with explicit `.status !== 0` checks compensating for `spawnSync`'s non-throwing failure mode. `node --check` passes on the extracted script body.
- The `update-index.ts` invocation now has an explicit `[STEP:deploy:FAIL:...]` / `[PIPELINE:FAIL]` branch matching the established convention.

However, adversarial tracing through `scripts/run.sh` under its own `set -euo pipefail` found a genuine, reproducible crash bug in code this same phase touched (the checksum-verification block), which can silently abort the script before it ever reaches the finish banner, the exit-code-based notification, or log rotation — i.e. it can defeat the exact reliability guarantee this phase set out to deliver. A related, pre-existing failure mode one line below (unguarded `git checkout --`) shares the same root cause and was not addressed. A secondary observability regression was found in `invest.md`'s deploy failure messaging, where three functionally distinct new failure causes (date-validation, commit, push) are now collapsed into one hardcoded, misleading message.

## Narrative Findings (AI reviewer)

### Critical Issues

#### CR-01: Checksum-lookup `grep -F | awk` pipeline can silently crash `run.sh` before it reports the real exit code

**File:** `scripts/run.sh:48` (inside the HTML-protection verification block, `scripts/run.sh:43-63`)

**Issue:** The script runs under `set -euo pipefail` (line 2). The line

```bash
EXPECTED=$(grep -F "$PROJECT_DIR/$f" "$CHECKSUM_FILE" | awk '{print $1}')
```

is a command substitution assigned to a plain variable. If `grep -F` finds no matching line in `$CHECKSUM_FILE` for the current `$f` (exit status 1), `pipefail` propagates that non-zero status to the whole pipeline, and since it is not guarded by `if`, `while`, `&&`/`||`, or an explicit `|| true`, `errexit` (`set -e`) terminates the **entire script** at that point — before the "Investment Pipeline Finished" banner is written, before the `EXIT_CODE`-based `terminal-notifier` branch runs, and before the 7-day log-rotation `find ... -delete` executes. This happens even when the actual `claude` pipeline run (the thing `EXIT_CODE` is supposed to describe) succeeded perfectly.

This is not a hypothetical: it reproduces with the *exact* code shape used in `run.sh`. A file is only ever checksummed at record-time (`scripts/run.sh:29-33`) if `[ -f "$PROJECT_DIR/$f" ]` was true *then*; but the verify-time loop (`scripts/run.sh:46-58`) independently re-checks `[ -f "$PROJECT_DIR/$f" ]` at the *end* of the run. Any protected file that did not exist when the checksum was recorded but exists by the time verification runs (e.g. a fresh checkout/bootstrap where `docs/portfolio.html` doesn't exist yet and gets created by this same pipeline run, or a file manually deleted between runs) hits this exact crash. Reproduced live:

```bash
$ bash -c 'set -euo pipefail
PROTECT_FILES=("docs/index.html" "docs/portfolio.html")
CHECKSUM_FILE=checksums.txt
touch docs/index.html                       # only index.html exists at "start"
for f in "${PROTECT_FILES[@]}"; do
  [ -f "$f" ] && shasum -a 256 "$f" >> "$CHECKSUM_FILE"
done
touch docs/portfolio.html                   # portfolio.html appears mid-run
for f in "${PROTECT_FILES[@]}"; do
  if [ -f "$f" ]; then
    EXPECTED=$(grep -F "$f" "$CHECKSUM_FILE" | awk "{print \$1}")
    echo "checked $f -> $EXPECTED"
  fi
done
echo "=== reached end of script ==="'
checked docs/index.html -> e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
$ echo $?
1
```

Note `=== reached end of script ===` never prints — the script dies silently at `docs/portfolio.html`'s lookup, with exit code 1, which has nothing to do with the real pipeline outcome.

Notably, the developer clearly was aware of this exact hazard elsewhere in the same diff: the sibling extraction at line 70 (`FAILED_STEP=$(grep -o ... | tail -1 | sed ...) || true`) is correctly guarded with `|| true` for precisely this reason. The `EXPECTED=` line — touched by this same phase's Task 3 (WR-01 fix, which only added the `-F` flag) — was not given the same protection.

**Fix:** Guard the pipeline the same way the `FAILED_STEP` line already does, and treat "no match" as a legitimate, non-fatal state:

```bash
EXPECTED=$(grep -F "$PROJECT_DIR/$f" "$CHECKSUM_FILE" 2>/dev/null | awk '{print $1}') || true
```

(or equivalently wrap the whole verification loop body with `set +e`/`set -e` bracketing). Add a regression test / manual check that simulates a protected file appearing only after the checksum snapshot, to prevent this from silently reappearing.

### Warnings

#### WR-01: Unguarded `git checkout --` in the same HTML-restore block can also abort the script under `set -e`

**File:** `scripts/run.sh:52`

**Issue:** `git -C "$PROJECT_DIR" checkout -- "$f"` has no error handling. If it fails for any reason (file not yet tracked by git, detached HEAD, corrupted index, permissions, etc.), `set -e` will abort the script at that exact line, again skipping the finish banner, `terminal-notifier` call, and log rotation — the same failure class as CR-01, and directly adjacent to the code this phase modified (the enclosing loop's header/PROTECT_FILES handling was touched by Task 3). This was not introduced by this diff, but it undermines the same OPS-03 reliability goal the phase exists to close, and was not caught by the phase's own verification.

**Fix:** At minimum, don't let a restore failure crash the whole script:

```bash
if ! git -C "$PROJECT_DIR" checkout -- "$f"; then
  echo "HTML保護: $f の復元に失敗しました" | tee -a "$LOG_FILE"
fi
```

#### WR-02: Deploy failure notification message is now inaccurate for 2 of its 3 possible causes

**File:** `.claude/commands/invest.md:1816-1820` (failure branch), interacting with `.claude/commands/invest.md:1766-1802` (deploy `node -e` script)

**Issue:** This phase's Task 2 (Plan 02) correctly introduced three distinct failure paths in the deploy `node -e` script, each with its own `console.error` message:
- `不正なdate形式: ' + date` (new, date-regex validation failure)
- `デプロイエラー: git commit に失敗しました` (commit failure)
- `デプロイエラー: git push に失敗しました` (push failure)

However, the instruction block that decides what to `echo` back into the pipeline log/notification for *any* non-zero exit of this script still hardcodes a single message that only describes the third case:

```bash
echo '[STEP:deploy:FAIL:git push に失敗]'
echo '[PIPELINE:FAIL] ステップ: deploy, エラー: git push に失敗'
```

So if the failure is actually a malformed `date` from the LLM-generated `meeting-result.json` (the exact vulnerability class this same plan was hardening against) or a `git commit` failure, the log/notification will still misleadingly say "git push に失敗" — directly reducing the diagnostic value of the `FAILED_STEP` notification feature this phase (Plan 01, Task 2) also just built. Since this phase explicitly touched the surrounding failure semantics of this exact script, the mismatch should have been reconciled at the same time.

**Fix:** Generalize the message to cover all three causes, e.g.:

```bash
echo '[STEP:deploy:FAIL:デプロイ処理が失敗（date検証/commit/pushのいずれか、詳細はログのconsole.error出力を参照）]'
echo '[PIPELINE:FAIL] ステップ: deploy, エラー: デプロイ処理が失敗'
```

### Info

#### IN-01: `spawnSync` failure paths discard the underlying error detail

**File:** `.claude/commands/invest.md:1790-1799`

**Issue:** `commitResult`/`pushResult` failure branches only log a fixed string (`デプロイエラー: git commit に失敗しました` / `... git push に失敗しました`), never surfacing `commitResult.error` / `pushResult.error` (populated when, e.g., the `git` binary itself is missing — `spawnSync` sets `.status === null` and `.error` in that case, per the plan's own Task 2 note). This makes root-causing a real production deploy failure harder than necessary.

**Fix:** Include the detail when present, e.g. `console.error('デプロイエラー: git commit に失敗しました' + (commitResult.error ? ' (' + commitResult.error.message + ')' : ''));`.

#### IN-02: Unused `hasChanges` variable in the deploy script (pre-existing, not touched by this diff)

**File:** `.claude/commands/invest.md:1776,1784`

**Issue:** `let hasChanges = false;` is declared and later set to `true` inside the `catch` block, but the variable is never read afterward — dead state that adds no behavior. Not introduced by this phase, but present in the reviewed file.

**Fix:** Remove the unused variable, or use it if it was meant to gate later logic.
