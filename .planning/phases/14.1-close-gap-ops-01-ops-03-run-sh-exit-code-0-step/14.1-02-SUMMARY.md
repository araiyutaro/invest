## Accomplishments
- `update-index.ts` invocation now has the same explicit STEP/PIPELINE FAIL-marker observability as every other pipeline step (data-collection, round-1/2/3, report-generation), closing the last unmonitored sub-action in the deploy step (D-08)

- Corrected a subtle semantic gap in the WR-04 review's suggested fix: `spawnSync` does not throw on nonzero child exit code (unlike `execSync`), so explicit `.status !== 0` checks were added after both the commit and push calls to preserve the pre-existing `[STEP:deploy:FAIL:git push に失敗]` detection contract

## Decisions Made
- Reused step id `deploy` for the new FAIL branch rather than introducing an `update-index` step id, per the plan's explicit interface guidance — `update-index.ts` is a sub-action of Step 4/deploy, not a separate pipeline step, so the existing single-`deploy`-step observability contract (`[STEP:deploy:START]` / `[STEP:deploy:OK]` / `[STEP:deploy:FAIL:*]`) is preserved with a second FAIL-reason variant.
- Added explicit `.status !== 0` checks on both `spawnSync` calls (commit and push) — this was flagged in the plan itself as a correction needed beyond the WR-04 review's suggested snippet, since `spawnSync` silently returns a nonzero `.status` on command failure rather than throwing, and without the check a failed `git commit`/`git push` would have been misreported as success.
