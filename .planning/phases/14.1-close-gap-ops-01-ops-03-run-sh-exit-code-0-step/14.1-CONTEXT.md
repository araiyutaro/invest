# Phase 14.1: Close gap: OPS-01/OPS-03 — run.sh EXIT_CODE常時0バグとSTEPマーカーのログ未到達を修正 - Context

**Gathered:** 2026-07-01
**Status:** Ready for planning

<domain>
## Phase Boundary

v2.3マイルストーン監査（`.planning/v2.3-MILESTONE-AUDIT.md`）で発見された、`scripts/run.sh`の単一箇所（39-41行目）に起因する2つの実運用バグを修正する。

1. **OPS-01（未達成）**: `claude -p "/invest"`のデフォルト出力フォーマット（text）がBashツール呼び出しの標準出力を破棄するため、invest.md内で`echo`されるSTEPマーカーが`logs/invest-*.log`に一切到達していない
2. **OPS-03（未達成）**: `>> "$LOG_FILE" 2>&1 || true`の`|| true`が`PIPESTATUS[0]`を無条件に0で上書きするため、`EXIT_CODE`が常に0になり、失敗時のmacOS通知（Basso音・異常終了表示）に到達できない

合わせて、監査が同一修正フェーズでの対応を推奨したシェルインジェクション修正（WR-04/WR-05相当）と、run.sh内の軽微な技術的負債（WR-01, WR-03）、およびdeployステップのFAILハンドリング欠如（WARNING2）を含める。

</domain>

<decisions>
## Implementation Decisions

### STEPマーカー到達方式 (OPS-01)
- **D-01:** `claude`呼び出しの出力フォーマットを`--output-format stream-json --verbose`に変更する。これによりツール呼び出し（Bashのecho出力含む）がストリームとして出力され、run.shの既存リダイレクトでログファイルに到達するようになる。invest.md内の既存`echo '[STEP:...]'`呼び出し（約18箇所）は変更不要
- **D-02:** ログファイルは生JSONL（1行1JSON）のまま保存する。追加の整形済みサマリファイルは作らない。障害調査時は`grep -o '\[STEP:[^]]*\]'`等でJSON文字列内のマーカーを抽出する運用とする
- **D-03:** run.sh側の既存ログ記録構造（`>> "$LOG_FILE" 2>&1`）はそのまま維持し、出力フォーマットフラグの追加のみ行う

### EXIT_CODE修正 (OPS-03)
- **D-04:** `|| true`による`PIPESTATUS`上書きを解消し、`claude`呼び出しの実際の終了コードを正しく捕捉する。`set -euo pipefail`によるスクリプト早期終了（HTML保護の検証・通知・ログローテーションが実行されなくなる）を避けつつ、正しい終了コードを保持する実装にする（`set +e`/`if`分岐等の具体的なbash実装手法は実装時に判断）

### 通知内容強化 (OPS-03)
- **D-05:** 失敗時のterminal-notifierメッセージに失敗ステップ名を含める。ログ内の最後の`[STEP:*:FAIL:...]`マーカーを抽出し、「パイプライン異常終了 (deployで失敗, exit: N)」のように表示する

### 修正スコープ
- **D-06:** WR-04/WR-05相当のシェルインジェクション修正を含める。対象は`.claude/commands/invest.md`のdeployステップで、LLM（moderator-final）が生成した`meeting-result.json`の`date`値をそのまま`execSync`の文字列に連結している箇所。date形式のバリデーション + `execSync`の引数配列化（またはシェル解釈を回避する方式）で対応する
- **D-07:** run.sh内の軽微な技術的負債を同時に修正する。WR-01: `grep`を`grep -F`に変更し固定文字列マッチにする（パスに`.`を含む場合の誤マッチ防止）。WR-03: `PROTECT_FILES`を配列化し、未クォート展開によるglob展開リスクを解消する
- **D-08:** `update-index.ts`呼び出し（invest.md 1746行付近、Phase 14で書き換えられたhero/accordion生成ロジック）に、他のStepと同様の明示的FAILハンドリングを追加する。例外時に`[STEP:deploy:FAIL:update-index.tsが失敗]`を出力してからパイプライン失敗サマリに進む
- **D-09:** WR-02（invest.md内でStep 3d がStep 3cより前に登場するドキュメント順序の不一致）は対象外。バグではなく可読性の問題のため、このフェーズでは扱わない

### 検証方針
- **D-10:** 実機ドライラン（`claude`呼び出しを意図的に失敗させてEXIT_CODEとログ到達を確認する）は必須としない。コードレビューでの確認で十分とする
- **D-11:** 修正実装後、次回のlaunchd自動実行（翌朝）で実際のログファイルを確認し、STEPマーカーとEXIT_CODEが正しく機能しているかをHuman-UAT項目として残す（プリマージのゲートにはしない、実運用での事後確認）

### Claude's Discretion
- `set +e`/`if`分岐等、EXIT_CODE捕捉の具体的なbash実装方法
- terminal-notifierメッセージの具体的な文言・フォーマット
- date値バリデーションの正規表現・エラー処理の詳細（execSync引数配列化 vs spawnSync等）
- 失敗ステップ抽出ロジック（`grep -o '\[STEP:[^]]*\]'`等）の実装場所（run.sh内かnode -eスクリプト内か）

</decisions>
