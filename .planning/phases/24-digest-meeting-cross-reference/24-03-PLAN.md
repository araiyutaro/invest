<objective>
Wire the Plan 01 matcher into the pipeline at `src/scripts/write-news-digest.ts`, computing the crossref map inside a dedicated, ISOLATED nested try/catch so a crossref exception can never flip the digest's exit code or trigger the null-fallback page. Add a dedicated `[STEP:digest-crossref:*]` marker to `invest.md` Step 3e. Implements XREP-02 fail-soft per D-13, D-14 and OPS-04.

Purpose: Guarantee that cross-reference computation is fail-soft — the digest and the existing 3 reports always generate/deploy even if crossref throws, with the failure visible only via an operator-facing STEP marker (never reader-facing, never [PIPELINE:FAIL]).
Output: isolated try/catch in write-news-digest.ts, new integration tests, dedicated STEP marker in invest.md.

LOCKED DESIGN DECISION (Open Question 1 / D-14): the STEP marker is signalled via **stderr string convention** (Option a) — write-news-digest.ts emits `console.error("[digest-crossref] OK")` on success and `console.error("[digest-crossref] FAIL: <reason>")` on the isolated catch; invest.md Step 3e inspects the just-run command output and echoes `[STEP:digest-crossref:OK]` or `[STEP:digest-crossref:FAIL:...]`. Chosen over the pipeline-metrics.json field option because it requires zero new file I/O and matches the existing convention where invest.md echoes markers based on the agent's observation of prior command output. The script NEVER prints a literal `[STEP:...]` string (verified codebase invariant: all STEP markers echo from invest.md only).
</objective>
