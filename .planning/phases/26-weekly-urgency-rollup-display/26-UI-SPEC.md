# Phase 26 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope reminder:** This phase adds exactly ONE new section to an existing static-HTML report (`docs/{date}/portfolio-report.html`). It is not a new page, not a new design system, and has zero client-side JS/interactivity — the report is generated once by `generatePortfolioReportHtml` (Node/TS) and served as static HTML. Every visual decision below **reuses** tokens/colors/classes already defined in `src/scripts/report-utils.ts` (`generateBaseStyles`) and `src/scripts/generate-portfolio-report.ts`. No new CSS file, no new class needed beyond what already exists (`.agent-card`, `h2`, `h4`, `p`), except where explicitly noted as new inline style.

## Spacing Scale

This codebase does **not** use a 4px-multiple token scale — `generateBaseStyles` and every existing renderer (`formatUrgentBadgeHtml`, `formatDecisionChangedBadgeHtml`, `formatHoldingEvaluationsHtml`, `formatHoldingNewsItemHtml`) use hand-tuned `rem` values directly in inline styles. To avoid introducing a second, inconsistent spacing system into the same file, this phase **reuses the existing rem values verbatim** rather than inventing a new 4px scale. Values below are the exact literals this section's markup must use (verified against live code, 16px root assumed, no root font-size override present).

| Token | Value | Usage (existing precedent) |
|-------|-------|-------|
| card padding | `1.2rem` (19.2px) | `.agent-card` padding (report-utils.ts L181) — the rollup section's cards inherit this class unchanged |
| card gap | `1rem` (16px) | `.agent-card` margin-bottom (report-utils.ts L182) — reused unchanged between per-symbol cards |
| symbol-card internal line gap | `0.2rem` (3.2px) top/bottom | NEW inline style on the urgent-flag `<p>` / decision-change `<p>` lines inside each symbol card — matches the tight vertical rhythm already used in `formatHoldingNewsItemHtml`'s sub-paragraph (`margin:0.15rem 0 0` — report-utils.ts-adjacent, generate-portfolio-report.ts L33) |
| footnote top margin | `0.4rem` (6.4px) | NEW inline style, matches `formatHoldingNewsItemHtml`'s sub-line top-margin convention |
| h2 top margin | `2rem` (32px) | `h2` margin-top (report-utils.ts L123) — reused unchanged, this section's `<h2>` gets no special-case margin |
| h2 bottom margin | `0.8rem` (12.8px) | `h2` margin-bottom (report-utils.ts L124) — reused unchanged |

Exceptions: none beyond the two NEW inline-style values listed above (`0.2rem` line gap, `0.4rem` footnote top-margin) — both are minimal, both mirror an existing analogous value elsewhere in the same file, neither introduces a new visual rhythm.

## Copywriting Contract

All strings below are FINAL and locked — planner/executor must use them verbatim (not paraphrase), so tests can assert exact string matches per RESEARCH.md's test map.

| Element | Copy |
|---------|------|
| Section heading | `今週の緊急・判断変更ロールアップ` |
| Primary CTA | not applicable — this is a read-only report section with no interactive elements (consistent with every other section in this report: `総括コメント`, `保有銘柄 個別評価`, `リバランス提案` also have no CTAs) |
| Empty state — Tier 1 (no history file / 0 total entries in `urgencyHistory`, i.e. `Object.keys(urgencyHistory).length === 0`) | `まだ緊急フラグ・判断変更の履歴がありません（履歴は日次で蓄積されます）` |
| Empty state — Tier 2 (history exists but zero movement in the 7-day window, i.e. `rollup.symbols.length === 0` while `Object.keys(urgencyHistory).length > 0`) | `今週は緊急フラグ・判断変更はありませんでした` |
| Partial state — Tier 3 footnote (appended whenever `rollup.daysCovered < 7`, regardless of whether Tier 2's empty message or the per-symbol card list is showing) | `（過去{daysCovered}日分の履歴に基づく）` — `{daysCovered}` is the literal integer from `WeeklyUrgencyRollup.daysCovered`, e.g. `（過去3日分の履歴に基づく）`. Never shown when `daysCovered === 7`. Never shown together with Tier 1 (Tier 1 implies `daysCovered === 0`, which would render a meaningless `過去0日分` string — guard explicitly against this). |
| Per-symbol urgent-flag line prefix | `⚠ 緊急フラグ: ` followed by ascending `MM/DD` dates joined with `", "` (halfwidth comma + space) — e.g. `⚠ 緊急フラグ: 07/02, 07/04`. Omitted entirely (no line rendered) when a symbol has zero urgent dates in-window. |
| Per-symbol decision-change line prefix | `判断変更: ` followed by `{MM/DD} {from} → {to}` — e.g. `判断変更: 07/03 保持 → 一部売却`. If a symbol has more than one decision-change event within the 7-day window, render one such line per event (separate `<p>`, chronological ascending order), each with the same `判断変更: ` prefix — do not comma-join multiple changes onto one line. |
| Error state | not applicable as a user-visible state — per D-14 fail-soft, any failure in history load/parse/aggregation degrades silently to the **Tier 1 empty-state copy above** (same rendering path, same string). There is no separate "something went wrong" message; this is an intentional design choice matching this report's existing fail-soft precedent (`本日のポートフォリオ分析は生成されませんでした。` on `portfolioAnalysis === null` is the only precedent for a degraded-state message anywhere in this file, and even that is a graceful message, not an error). |
| Destructive confirmation | not applicable — zero destructive actions in this phase (read-only aggregation display) |

## Per-Symbol Row Layout (locked structure)

One `<div class="agent-card">` per symbol with movement (mirrors the existing per-holding card pattern in `formatHoldingEvaluationsHtml`, generate-portfolio-report.ts L82-87 — reuse the class, not a new card style). No border-left color override (unlike individual holding cards, a rollup entry has no single "current decision," so it uses the plain default `.agent-card` border-left `#6366f1`, unchanged — do not attempt to derive a decision color for the card border).

```html
<div class="agent-card">
  <h4>{escapeHtml(symbol)}{nameJa ? " -- " + escapeHtml(nameJa) : ""}</h4>
  <!-- only rendered if urgentDates.length > 0 -->
  <p style="color:#ef4444;font-size:0.9rem;font-weight:bold;margin:0.2rem 0;">⚠ 緊急フラグ: {joined MM/DD dates}</p>
  <!-- one <p> per decision-change event, only rendered if decisionChanges.length > 0 -->
  <p style="color:#f59e0b;font-size:0.9rem;font-weight:bold;margin:0.2rem 0;">判断変更: {MM/DD} {escapeHtml(from)} → {escapeHtml(to)}</p>
</div>
```

Symbols are rendered in the order already returned by `computeWeeklyUrgencyRollup` (deterministic `symbol.localeCompare` ascending order per RESEARCH.md Pattern 1 — do not re-sort in the renderer).

### Full section render logic (locked decision tree)

```
totalHistoryEntries = Object.keys(urgencyHistory).length   // raw history, NOT window-filtered

if totalHistoryEntries === 0:
  render: <h2>今週の緊急・判断変更ロールアップ</h2>
          <div class="agent-card">
            <p style="color:#888;font-size:0.85rem;">まだ緊急フラグ・判断変更の履歴がありません（履歴は日次で蓄積されます）</p>
          </div>

else if rollup.symbols.length === 0:
  render: <h2>今週の緊急・判断変更ロールアップ</h2>
          <div class="agent-card">
            <p style="color:#888;font-size:0.85rem;">今週は緊急フラグ・判断変更はありませんでした</p>
            <!-- only if rollup.daysCovered < 7 -->
            <p style="color:#888;font-size:0.85rem;margin-top:0.4rem;">（過去{daysCovered}日分の履歴に基づく）</p>
          </div>

else:
  render: <h2>今週の緊急・判断変更ロールアップ</h2>
          {one .agent-card per symbol, per layout above, in rollup.symbols order}
          <!-- only if rollup.daysCovered < 7, as a trailing plain <p>, NOT inside a card -->
          <p style="color:#888;font-size:0.85rem;margin-top:0.4rem;">（過去{daysCovered}日分の履歴に基づく）</p>
```

**Placement (D-08, locked):** Insert this entire block immediately after `overallCommentHtml` and immediately before `holdingEvaluationsHtml` in the main template composition (generate-portfolio-report.ts, current L147-149).

**Open question flagged by RESEARCH.md (Pitfall 4) — recommend planner lock this explicitly:** The `portfolioAnalysis === null` early-return branch (L110-129) currently renders none of the other sections. Because this rollup's only data dependency is `urgencyHistory` + `result.date` (not `portfolioAnalysis`), the recommended behavior is to render this section in **both** branches (null and non-null), reinforcing the fail-soft story. This is a research recommendation, not a CONTEXT.md-locked decision — planner must make an explicit go/no-go call and, if adopted, this section's HTML must appear in the null-analysis branch's template too, in the same visual form specified above.

**Date formatting (D-07, locked):** All dates rendered as `MM/DD` via pure string slicing of the canonical `YYYY-MM-DD` key (`dateKey.slice(5).replace("-", "/")`) — **do not** reuse `formatPublishedAtJst` (it is built for full ISO datetimes, adds `HH:MM`, and does not zero-pad month/day — see RESEARCH.md Pitfall 3). No timezone conversion is applied to date-only keys; internal aggregation and display both treat the key as already-canonical JST calendar dates.

**Escaping (locked):** `symbol`, `nameJa`, and `decision` enum values (`from`/`to`) MUST pass through `escapeHtml` before interpolation, matching every other dynamic string in this file. `MM/DD` date strings are derived internally via pure string slicing of an `isValidDateKey`-validated key and contain no HTML-significant characters by construction; escaping them is not required for safety, but doing so anyway (for uniform code style) is acceptable and not prohibited.
