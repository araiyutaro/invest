# Phase 31 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable — no component framework in this project |
| Component library | none — hand-rolled HTML string templates generated server-side (Node.js/TypeScript), no React/Vue/Next/Vite. Confirmed via `ls components.json tailwind.config.* package.json` — no `components.json`, no Tailwind, no React/Next/Vite in `package.json` |
| Icon library | none — no icon glyphs used anywhere in existing reports; visual differentiation is achieved entirely through color, badge shape (pill span), and typographic weight. This phase MUST NOT introduce icon glyphs (no emoji, no SVG icon set) — stay consistent with existing `formatUrgentBadgeHtml`/`formatDecisionChangedBadgeHtml` precedent which uses text labels only (the one exception, `⚠`, is an existing literal character already baked into `formatUrgentBadgeHtml`; do not add new ones in this phase) |
| Font | `'Helvetica Neue', Arial, 'Hiragino Sans', 'Hiragino Kaku Gothic ProN', Meiryo, sans-serif` (from `generateBaseStyles`, `report-utils.ts:106`) — do not override |

This project renders static HTML reports (Bloomberg-style dark theme) deployed to GitHub Pages. There is no client-side interactivity, no build step for CSS, and no component registry. All styling is inline `style=""` attributes or the shared `<style>` block emitted by `generateBaseStyles("#3b82f6")` (already called once per report in `generateDailyReportHtml`). This UI-SPEC therefore documents the **existing embedded design tokens** this phase must reuse, not a new design system to establish.

## Spacing Scale

Declared values (must be multiples of 4, matching existing inline styles in `generate-portfolio-report.ts`/`report-utils.ts`):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (0.25rem) | Badge internal vertical padding (`0.15rem`–`0.2rem` ≈ 2.4–3.2px, existing badge precedent — treat as the project's established xs unit even though it is not an exact 4px multiple; do not introduce a new value) |
| sm | 8px (0.5rem) | Badge `margin-left`, gap between ticker/company-name and badges, badge horizontal padding (`0.5rem`–`0.6rem`) |
| md | 16px (1rem) | `.agent-card` padding (`1.2rem` existing precedent — reuse exactly, do not redesign), paragraph `margin-bottom` |
| lg | 24px | Not used in this phase's card content (reserved for section-level spacing already handled by `<hr>`/`<h2>` in `generateBaseStyles`) |
| xl+ | — | Not applicable — this phase adds one section between two existing sections; page-level spacing is already governed by `generateBaseStyles` and untouched |

**Exceptions (must match, do not deviate):** All badge/pill/meta spacing values in this phase MUST byte-for-byte match the existing values copied from `formatUrgentBadgeHtml` / `formatDecisionChangedBadgeHtml` (`generate-portfolio-report.ts`) per RESEARCH.md Pitfall 4. Specifically: badge padding `0.15rem 0.5rem` (change/urgent-style badges) or `0.2rem 0.6rem` (primary buy badge, slightly larger to convey prominence per D-09), `margin-left: 0.5rem`, `border-radius: 999px` (pill shape), `font-size: 0.75rem`–`0.8rem`. Card container reuses `.agent-card` class as-is (padding `1.2rem`, `border-radius: 8px`, `margin-bottom: 1rem`) — no new CSS class for the card wrapper.

## Color

This project's Daily Report already commits to a locked accent (`generateBaseStyles("#3b82f6")`, blue, called once in `generateDailyReportHtml` — do not call it again or change the argument). This phase's new section inherits that accent and additionally reuses the portfolio-report badge palette per D-09/D-10 (verbatim color values, verified from `generate-portfolio-report.ts`):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f0f1a` (body background) | Page background — inherited, not touched by this phase |
| Secondary (30%) | `#1e1e2e` (`.agent-card` background) | Watchlist judgment card background — reuse `.agent-card` class as-is |
| Accent (10%) | `#3b82f6` / `#a5b4fc` (Daily Report's locked accent + `.agent-card h4` heading color) | `h2` section heading left-border and color (inherited from `generateBaseStyles`), card `h4` ticker/company-name text color (inherited `.agent-card h4` rule — do not override) |
| Buy badge (primary signal) | `#10b981` background, `#0f0f1a` text | **Reserved exclusively for** the `todayAction: "buy"` badge ("今日買うべき"). This is the ONE prominent, attention-grabbing color in the new section — per D-09 "buy signal alone should jump out visually" |
| Wait label (subdued) | `#9ca3af` text, no background (not a pill) | **Reserved exclusively for** the `todayAction: "wait"` label. Deliberately NOT styled as a badge/pill — plain inline text only, to create asymmetric visual weight vs. the buy badge (D-09) |
| Change — new signal (wait→buy) | `#10b981` background, `#0f0f1a` text | **Reserved exclusively for** `actionChanged === true` badge when `previousAction === "wait" && todayAction === "buy"` ("シグナル点灯: 待ち → 買い") |
| Change — downgrade (buy→wait) | `#f59e0b` background, `#1a1a28` text | **Reserved exclusively for** `actionChanged === true` badge in the opposite direction ("買い → 待ち") — same amber family as existing `formatDecisionChangedBadgeHtml` |
| Skipped / gray | `#4b5563` (border-left), `#6b7280` (label text), card `opacity: 0.7` | **Reserved exclusively for** `status: "skipped"` cards ("判定不能（データ不足）") — must be visually distinct from both the "wait" label color (`#9ca3af`) and the "buy" badge, per D-07 |
| Muted meta text | `#888` | **Reserved exclusively for**: as-of timestamp + market session note (D-06), `addedDate` "登録日" line (D-08), the empty-state one-liner "現在ウォッチリスト銘柄はありません" (D-14 state ②) |
| Destructive | not applicable | This phase has no destructive actions — decision-support display only (per CONTEXT.md §Specific Ideas: "判定は decision-support であり執行指示ではない") |

**Accent reserved for:** `#10b981` (green) is reserved for exactly two things — the "buy" badge and the "new signal" change badge — never used elsewhere in this section. `#f59e0b` (amber) is reserved for exactly one thing — the "downgrade" change badge — matching the existing `formatDecisionChangedBadgeHtml` amber, preserving cross-report color consistency (portfolio-report's judgment-changed badges use the same amber). No other new colors are introduced by this phase.

## Copywriting Contract

All copy is Japanese (project convention — 100% of existing report UI text is Japanese). Locked per 31-CONTEXT.md D-01–D-14 and RESEARCH.md Pattern 2–4 (values below are the concrete strings the executor must use verbatim unless a Claude's-Discretion override is explicitly noted):

| Element | Copy |
|---------|------|
| Section heading | `ウォッチリスト 買いタイミング判定` (h2, RESEARCH.md Pattern 4 — Claude's Discretion per CONTEXT.md, this exact string is the recommended default and should be used unless plan-time discussion changes it) |
| Section subhead (non-empty state only) | `ウォッチリスト銘柄の本日の買いタイミング判定です。` |
| Buy badge (primary CTA-equivalent — decision-support label, not a clickable action) | `今日買うべき` |
| Wait label | `待ち` |
| Change badge — new signal (wait→buy) | `シグナル点灯: 待ち → 買い` |
| Change badge — downgrade (buy→wait) | `買い → 待ち` |
| Skipped status label | `判定不能（データ不足）` |
| As-of session note — US market | `前日終値時点` |
| As-of session note — JP market | `寄付き前時点` |
| Registered-date meta prefix | `登録日: {addedDate}` |
| Empty state (state ② — file valid, zero active judgments) | `現在ウォッチリスト銘柄はありません` (single line, no heading beyond the section h2 itself — matches locked D-14 wording verbatim) |
| Section omitted (state ③ — loader null: missing/corrupt/stale-date) | *(no copy — section renders as empty string, entirely absent from the page; this is intentional per D-14, not an error state shown to the reader)* |
| Card heading fallback (no company name resolvable) | `{ticker}` alone, no em-dash suffix (D-04 fallback) |
| Destructive confirmation | not applicable — no destructive actions in this phase (read-only decision-support display) |

**Copy rules carried from project convention (apply to all dynamic values in this section):**
- All LLM-derived or file-derived strings (`rationale`, `signals[]`, `ticker`, `nameJa`/`name`, `addedDate`, `asOf`) MUST pass through `escapeHtml` (`report-utils.ts`) before interpolation — no exceptions (XSS defense-in-depth, RESEARCH.md Security Domain).
- `rationale` is rendered as prose in a `<p>`, never a table (D-03, established v2.1 convention).
- Do not paraphrase or invent copy not listed above; where CONTEXT.md marks a string as Claude's Discretion (section heading wording, pill vs. bullet-list for signals), the values in this contract are the locked defaults for `--auto` mode.

**Signals rendering — locked choice (researcher's decision per CONTEXT.md D-05 discretion grant):** Render `signals[]` as **inline pills** reusing the existing `.ticker-pill` CSS class verbatim (`report-utils.ts:194-204` — background `#2a2a3e`, color `#c4b5fd`, `font-size: 0.8rem`, `padding: 0.15rem 0.5rem`, `border-radius: 999px`, `border: 1px solid #3f3f5a`). Rationale for choosing pills over a bullet list: (1) `.ticker-pill` already exists and is proven across three prior reports (news-digest, portfolio-report crossref chips, daily-report ticker mentions) — zero new CSS; (2) signals are short confluence-gate labels (e.g. "MA上抜け", "出来高急増"), which read better as compact scannable chips than as a vertical `<ul>` that would visually compete with the rationale paragraph directly above it; (3) matches RESEARCH.md Pattern 3's own example code which already uses `.ticker-pill` for this exact field. Do not introduce a new pill variant class — reuse `.ticker-pill` exactly as-is (do not fork into a `.signal-pill` class).
